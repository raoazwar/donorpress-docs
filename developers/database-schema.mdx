DonorPress creates seven custom tables on activation. They're prefixed with `wp_donorpress_` (or your WP table prefix).

## `wp_donorpress_donors`

One row per donor. Identity key: `email`.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | BIGINT UNSIGNED, AUTO_INCREMENT, PK | Internal donor ID. |
| `email` | VARCHAR(190) UNIQUE | Identity key. |
| `first_name` | VARCHAR(100) | Optional. |
| `last_name` | VARCHAR(100) | Optional. |
| `phone` | VARCHAR(50) | Optional. |
| `company` | VARCHAR(150) | Optional. |
| `total_donated` | DECIMAL(15,2), DEFAULT 0 | Lifetime total of completed donations. |
| `donation_count` | INT UNSIGNED, DEFAULT 0 | Lifetime count of completed donations. |
| `largest_donation` | DECIMAL(15,2), DEFAULT 0 | Single largest donation. |
| `avg_donation` | DECIMAL(15,2), DEFAULT 0 | Average donation. |
| `is_anonymous` | TINYINT(1), DEFAULT 0 | Public anonymity flag. |
| `user_id` | BIGINT UNSIGNED, NULL | Linked WordPress user ID. |
| `date_created` | DATETIME | First seen. |
| `date_modified` | DATETIME | Last update. |

Indexes: PK on `id`, UNIQUE on `email`, INDEX on `user_id`.

## Migrations

The migrator (`src/Database/Migrator.php`) tracks schema version in the `donorpress_db_version` option. On every plugin update:

1. Compare stored version to current.
2. Run any pending migrations via `dbDelta()`.
3. Update the stored version.

If you ever need to force a re-run, use **Settings â†’ Advanced â†’ Run migrations**.
